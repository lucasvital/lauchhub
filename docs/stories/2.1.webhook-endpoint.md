## Story

**As a** Kiwify enviando eventos de venda,
**I want** um endpoint público que aceite o token da campanha na URL, valide, enriqueça o payload e enfileire jobs nos workers ativos,
**so that** cada evento dispare automação cross-channel sem o Kiwify precisar saber de nada além da URL.

## Acceptance Criteria

1. Rota `POST /webhook/:campaign_token` registrada em `src/gateway/routes/webhook.ts`.
2. Lookup da campanha em Postgres por `campaign_token`:
   - **Não encontrada** → grava em `unmatched_events` (payload + token) e retorna 200.
   - **Encontrada e inativa** → ignora silenciosamente, retorna 200 (não enfileira nada).
   - **Encontrada e ativa** → segue fluxo de enrich + enqueue.
3. Detecção do tipo de evento Kiwify a partir do payload (campos `order_status`, `event` ou equivalente). Mapear pelo menos os 8 eventos listados em CLAUDE.md → seção "Eventos Kiwify".
4. Enrich do payload combinando dados Kiwify + config da campanha (tags por evento, segment_id, template_id, sheets_id, etc.).
5. Para cada worker presente em `campaign.enabled_workers[event_id]`, adicionar job na fila correspondente (`queue:sheets`, `queue:chatwoot`, `queue:mautic`, `queue:meta`).
6. **Resposta sempre 200** ao Kiwify — mesmo em erro interno (DB fora, Redis fora). Erro é logado com `correlation_id`, nunca propagado ao cliente.
7. Cada job tem `correlation_id` único para tracing cross-worker.
8. Validação básica do payload: rejeita payload vazio ou sem `Customer.email` E sem `Customer.mobile` (loga `validation_failed`, ainda retorna 200).
9. Logs estruturados (pino) em cada etapa: `webhook_received`, `campaign_matched`/`campaign_unmatched`, `jobs_enqueued`.
10. Testes de integração cobrem: match → enqueue, unmatched → save, DB down → still 200, Redis down → still 200, payload inválido → still 200.

## Dev Notes

**Referência primária:** CLAUDE.md → seção "Gateway" (snippet TypeScript já está lá, usar como base).

**Payload de referência Kiwify:** CLAUDE.md → "Payload de referência (Kiwify)".

**Tabela de eventos → workers padrão:** CLAUDE.md → "Eventos Kiwify". Mas atenção: a tabela são os **defaults**. O que vale é `campaign.enabled_workers[event_id]` (override por campanha, configurável no painel).

**Detecção de evento:**
- `order_status === 'paid'` → `compra_aprovada`
- `order_status === 'abandoned' || 'abandoned_cart'` → `carrinho_abandonado`
- `order_status === 'pix_generated'` → `pix_gerado`
- `order_status === 'billet_generated'` → `boleto_gerado`
- `order_status === 'refused'` → `compra_recusada`
- `order_status === 'refunded'` → `compra_reembolsada`
- `webhook_event_type === 'subscription_canceled'` → `subscription_canceled`
- `webhook_event_type === 'subscription_renewed'` → `subscription_renewed`

(Validar com payload real do Kiwify ao implementar — pode haver variação.)

**Estrutura sugerida do job (`WebhookJob`):**
```ts
{
  correlation_id: string,
  campaign_id: string,
  campaign_token: string,
  event: EventId,
  worker: WorkerId,  // 'sheets'|'chatwoot'|'mautic'|'meta'
  contact: { name, email, phone },
  order: { id, ref, status, payment_method, value, product_id, product_name },
  config: { /* slice da campanha relevante pro worker */ },
  received_at: string,
}
```

**Anti-pattern a evitar:** NÃO retornar 4xx/5xx pro Kiwify mesmo em "payload malformado". Kiwify para de enviar webhooks após muitas falhas.
