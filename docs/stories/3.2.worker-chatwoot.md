## Story

**As a** gestor de relacionamento,
**I want** cada evento de campanha criar/atualizar contato no Chatwoot com labels específicas (sem perder labels prévias),
**so that** o time de suporte veja o contexto completo da jornada do aluno direto no painel.

## Acceptance Criteria

1. `src/workers/chatwoot.worker.ts` cria BullMQ Worker consumindo `queue:chatwoot`.
2. Sequência implementada **exatamente** como em CLAUDE.md:
   - GET `/api/v1/accounts/:id/contacts/search?q={phone}`
   - Se não encontrado → POST `/api/v1/accounts/:id/contacts` com inbox_id da campanha
   - GET `/api/v1/accounts/:id/contacts/:cid/labels` (labels atuais)
   - **MERGE** labels atuais ∪ novas labels (sem duplicar)
   - POST `/api/v1/accounts/:id/contacts/:cid/labels` com a lista mesclada
3. ⚠ **Crítico:** API de labels sobrescreve — NUNCA fazer POST sem ler antes (sob risco de apagar labels manualmente adicionadas por atendentes).
4. Auth via `api_access_token` no header `api_access_token`.
5. Base URL e account_id vêm do `global_config` (carregados no startup ou cache curto).
6. Tags a aplicar = `campaign.chatwoot_tags[event_id]` (array).
7. Telefone normalizado: aceita `41999999999` ou `+5541999999999`, normaliza pra `+5541999999999`.
8. Retry policy 4 tentativas com backoff `[0s, 30s, 5min, 30min]`.
9. Erros classificados:
   - 401 → fail-fast pra DLQ
   - 404 contato deletado entre GET e POST → re-cria via POST contact + retry
   - 5xx → retry
10. Logs estruturados com `correlation_id`.
11. Testes (vitest + nock) cobrem: contato novo cria + labels, contato existente merge, race condition (404 entre passos), 401 fail-fast.
