## Story

**As a** marketer,
**I want** cada evento criar/atualizar contato no Mautic com tags e adicionar ao segmento da campanha,
**so that** automations de email/SMS no Mautic disparem automaticamente.

## Acceptance Criteria

1. `src/workers/mautic.worker.ts` cria BullMQ Worker consumindo `queue:mautic`.
2. Sequência **exatamente** como em CLAUDE.md:
   - GET `/api/contacts?search=email:{email}`
   - Se não encontrado → POST `/api/contacts/new`
   - PATCH `/api/contacts/:id/edit` aplicando tags
   - POST `/api/segments/:segId/contact/:cid/add`
3. Auth OAuth2 (`client_credentials` grant) com `MAUTIC_CLIENT_ID` + `MAUTIC_CLIENT_SECRET`.
4. Token cacheado em memória com refresh 60s antes do expires_at.
5. Tags a aplicar = `campaign.mautic_tags[event_id]`.
6. Segmento = `campaign.mautic_segment_id` (mesmo segment para todos os eventos da campanha por enquanto).
7. Tags são **acumulativas** (merge com tags existentes — Mautic já faz isso nativamente via PATCH, validar comportamento).
8. Retry policy padrão (4 tentativas, backoff exponencial).
9. Erros classificados:
   - 401 token expirado → renovar token + retry sem incrementar attempt counter (1x)
   - 401 credenciais inválidas → fail-fast pra DLQ
   - 5xx → retry
10. Logs estruturados.
11. Testes cobrem: novo contato + tags + segment, existing contact + tags merge, token refresh on 401, fail-fast em credencial errada.

## Dev Notes

**Referência:** CLAUDE.md → seção "Workers → Mautic Worker".

**OAuth2 detalhe:** Mautic suporta `client_credentials` grant (cliente confidencial). Endpoint: `POST {MAUTIC_URL}/oauth/v2/token` com body `grant_type=client_credentials&client_id=...&client_secret=...`.

**Tag merge:** Mautic permite tags incrementais via campo `tags`: `"tags":["tag1","tag2"]` na PATCH — comportamento depende da versão. Validar em ambiente real. Plan B: GET tags atuais, merge, PATCH com array completo.

**Diferença vs Chatwoot:** Search key aqui é **email**, não phone (mais comum em CRM).
