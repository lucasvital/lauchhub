## Story

**As a** Lucas (stakeholder do LaunchHub),
**I want** backend rodando em `https://launches.loyoladigital.com` (Coolify) e painel em `https://launchhub.vercel.app` (ou domínio custom), com webhooks Kiwify funcionando,
**so that** as campanhas reais parem de depender de scripts manuais e passem a fluir automaticamente.

## Acceptance Criteria

### Backend (Coolify no VPS)

1. DNS `launches.loyoladigital.com` apontando pro IP do VPS Coolify (A record).
2. App configurada no Coolify usando o `docker-compose.yml` da Story 6.1 (build pack: Docker Compose).
3. Domínio plugado na app via UI do Coolify → SSL Let's Encrypt automático.
4. Env vars completas configuradas no Coolify UI (todas as do `.env.example`, marcadas como secret onde apropriado).
5. `https://launches.loyoladigital.com/health` retorna 200 com TLS válido.
6. Migrations aplicadas em produção (via terminal do Coolify ou init job).
7. Volumes Postgres + Redis persistindo dados entre redeploys.
8. `CORS_ALLOWED_ORIGIN` env aponta pro domínio Vercel do frontend.

### Frontend (Vercel)

9. Repo conectado no Vercel (auto-detect Vite).
10. Env var `VITE_API_BASE_URL=https://launches.loyoladigital.com` em Production e Preview.
11. `https://<projeto>.vercel.app` carrega painel sem erros de console.
12. Login no painel funciona end-to-end (request `/api/login` → cookie → fetch das campanhas).
13. (Opcional) Domínio custom no Vercel (ex: `painel.loyoladigital.com`) — registrar e plugar.

### Integração & smoke E2E

14. Webhook configurado no painel Kiwify com URL `https://launches.loyoladigital.com/webhook/<token>` pra campanha de teste.
15. Pelo menos 1 campanha real cadastrada via painel (sugestão: `dg-pg02` ou `bbe-pr1`).
16. Teste de ponta-a-ponta — disparar evento Kiwify (sandbox ou compra real R$ 1) e validar:
    - Linha apareceu na Sheets
    - Contato criado/atualizado no Chatwoot com labels corretas
    - Contato no segmento certo no Mautic
    - Template HSM enviado no WhatsApp do número de teste
17. Backup automático do Postgres configurado (`pg_dump` diário → B2/S3 ou cron container).
18. Runbook em `docs/runbook.md`: restart, rollback, ler logs, rotacionar token, restaurar backup.

## Dev Notes

**Mudança vs CLAUDE.md original:** O CLAUDE.md prevê Docker Swarm com Traefik manual + app monolítica servindo painel. Decisão arquitetural 2026-05-14: **separar deploy** — backend no Coolify (VPS existente, gerencia Traefik internamente), frontend no Vercel (Vite estático).

**Por que essa separação:**
- Coolify já está no VPS, Docker-first, gerencia SSL/proxy
- Vercel é nativo pra Vite, preview por branch, deploy automático no push
- Frontend é admin-only (só Lucas) — não justifica CDN global, mas a ergonomia de dev (preview URL, rollback 1-click) compensa

**Pré-requisitos operacionais (não-código):**
- VPS com Coolify rodando ✓ (já existe)
- Conta Vercel (gratuita serve)
- DNS sob controle (Cloudflare/Registro.br)
- Domínio `launches.loyoladigital.com` resolvendo no IP do VPS
- Portas 80/443 abertas no VPS

**Rollback strategy:**
- **Backend (Coolify):** UI tem botão "Rollback to previous deployment" — instantâneo. Volumes preservam dados.
- **Frontend (Vercel):** Aba "Deployments" → "Promote to Production" no deploy anterior. Zero downtime.

**Variáveis env em produção:**
- Coolify UI gerencia (não commitar `.env`)
- Vercel: dashboard → Environment Variables → diferenciar Production/Preview/Development

**Secrets ROTATION:**
- `META_TOKEN`, `CHATWOOT_TOKEN`, `MAUTIC_CLIENT_SECRET` — rotacionar a cada 90 dias
- Atualizar no Coolify → restart automático do app

### Testing

- Smoke tests pós-deploy:
  - `curl -i https://launches.loyoladigital.com/health` → 200, TLS válido
  - `curl -i -X POST https://launches.loyoladigital.com/webhook/test-token` → 200 (vai pra unmatched_events)
  - Abrir Vercel URL → painel carrega → login → lista de campanhas vazia OK
- E2E manual (AC 16) é o teste definitivo
- Falha de smoke → rollback manual (MVP — automação fica pra hardening pós-MVP)
