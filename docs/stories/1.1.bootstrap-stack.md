## Story

**As a** desenvolvedor do LaunchHub,
**I want** uma base Node.js com Fastify, BullMQ e Postgres já conectados e funcionais,
**so that** as stories seguintes (gateway, workers, painel) tenham infraestrutura sobre a qual construir.

## Acceptance Criteria

1. `package.json` na raiz com Node 18+ e scripts: `dev`, `build`, `start`, `lint`, `typecheck`, `test`, `migrate`.
2. TypeScript configurado (strict mode, ESM, target ES2022).
3. Estrutura de pastas exatamente como em CLAUDE.md → seção "Estrutura de Pastas" (`src/gateway/`, `src/workers/`, `src/queue/`, `src/db/`, `src/ui/` já existe, `src/config.ts`).
4. `src/config.ts` lê variáveis do `.env` validando com zod/envalid (DATABASE_URL, REDIS_URL, PORT, NODE_ENV, e credenciais por serviço).
5. `src/gateway/server.ts` sobe servidor Fastify em `PORT` e responde 200 em `GET /health` com `{ ok: true, version, uptime }`.
6. `src/db/index.ts` exporta pool Postgres (pg ou postgres.js) conectando via `DATABASE_URL`. `GET /health` valida `SELECT 1`.
7. `src/queue/index.ts` configura conexão Redis para BullMQ e exporta as 4 queues (`sheets`, `chatwoot`, `mautic`, `meta`) — ainda sem workers consumindo.
8. `npm run dev` (com Postgres + Redis disponíveis localmente) sobe sem erros.
9. README na raiz documenta setup local (clonar, `npm i`, `.env.example` → `.env`, subir Postgres+Redis via docker, `npm run dev`).

### Completion Notes List

- All 9 acceptance criteria implemented and validated locally (typecheck + lint + 3 vitest specs all green).
- Used `pg` driver (not `postgres.js`) per Dev Notes recommendation.
- ioredis `Redis` named import (not default) — default import was rejected by TS under `module: NodeNext`. Documented inline.
- `META_API_VERSION` added to `.env.example` and zod schema with default `v20.0` (PO Cross-cutting item C2).
- `WebhookJob` type defined in `src/types/job.ts` with all `WorkerId`/`EventId` enums for downstream stories (PO Cross-cutting item C3).
- `migrate` npm script is a placeholder that exits 0 — real implementation lands in Story 1.2.
- BullMQ defaults: 4 attempts, exponential backoff starting at 30s (matches CLAUDE.md retry policy: 1 → 30s → 5min → 30min → DLQ).
- Health endpoint returns 503 when DB or Redis are down — chosen over 200 because `/health` is for internal monitoring, not Kiwify webhooks. Kiwify-facing 200-always behavior lives in Story 2.1's gateway route.
- Docker E2E smoke test (manual `curl /health` against live containers) NOT executed — Docker Desktop engine offline in this environment. Vitest mocks cover the same logic. README documents the manual validation steps.
- No git operations performed (project is greenfield, no repo initialized). When the user wants to push, activate `@devops` with `*environment-bootstrap` first.
